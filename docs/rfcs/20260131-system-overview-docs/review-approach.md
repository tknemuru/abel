P0 該当なし。本 RFC の方向性・スコープ・前提に重大な論理的欠陥は認められない。

### 2. 良い点 (Strengths)

- **2. 背景・動機**: 課題定義が正確である。CLAUDE.md の規約と現状のギャップを表形式で明示しており、RFC の必然性に疑問の余地がない。`/rfc`・`/imp` 等のワークフローコマンドが規約通りのファイル名を前提としている点も、単なるドキュメント整理ではなくツールチェーンとの整合性確保という実務的な動機として妥当である。

- **6. 代替案の検討（案B の選定）**: `ops.md` と `learning-data-contract.md` を統合対象外とする判断は合理的である。実際に `ops.md` の内容（セットアップ手順、トラブルシューティング、バックアップ等）は IEEE 1016 の Structure / Data / Interface いずれにも自然に分類できず、無理に押し込めば各ドキュメントの凝集度が低下する。`learning-data-contract.md` も abel-learning との契約仕様という独立した文書目的を持ち、`api-overview.md` に統合すると参照性が著しく低下する。案A・案C の短所分析も的確である。

- **5.1 マッピング表**: 既存ドキュメントの全セクションについて移行先を網羅的に定義しており、実装時の判断のブレを防いでいる。特に「不明点/要確認リスト」を明示的に移行対象外とした判断は、新ドキュメントに古い未解決項目を持ち込まないという意図が明確で妥当である。

- **5.2 [仮定] セクション**: abel が REST API サービスではなく CLI ツールである特性を踏まえ、`api-overview.md` の「エンドポイント」を CLI コマンド、「認証方式」を外部サービス接続方式と読み替える判断を明示している。CLAUDE.md の「省略可」条項に対して「省略しない」という判断とその根拠も記載されており、設計判断の透明性が高い。

P0: 該当なし

**[P1-1] spec.md「不明点/要確認リスト」の一部項目は移行を検討すべき**
- **対象セクション**: 5.1 既存ドキュメントと新ドキュメントの対応関係（spec.md マッピング表）
- **内容**: `spec.md` の「不明点/要確認リスト」を一律移行しない方針としているが、4項目のうち一部は新ドキュメントに残す価値がある可能性がある。例えば「`resources/configs/index.yml` の必須キー」は `ops.md` に既に詳細が記載されているため問題ないが、「cron/スケジューラ設定」の項目は運用上の未解決事項として `ops.md` の課題セクション等に記載する価値があり得る。ただし本 RFC のスコープは `ops.md` の改廃を含まないため、別途対応でも問題ない。
- **修正の期待値**: 移行しない項目の中で他ドキュメントに引き継ぐべきものがないか、実装時に一度精査することを推奨する。必須ではない。

**[P1-2] architecture.md に「デプロイ構成」が含まれていない**
- **対象セクション**: 5.2 新ドキュメントの構成（docs/architecture.md）
- **内容**: CLAUDE.md の `architecture.md` の記載内容に「デプロイ構成」が含まれているが、5.2 の architecture.md 構成案にはデプロイ構成に相当するセクションが存在しない。abel は CLI ツールでありデプロイ構成が単純（ローカル実行 + WSL 環境）であるため、大きなセクションにはならないが、規約との整合性の観点から、技術スタックのセクション内等に簡潔に記載しておくことが望ましい。
- **修正の期待値**: `architecture.md` の構成案に、デプロイ構成（実行環境: ローカル/WSL、外部依存サービス等）に関する簡潔な記述を追加することを推奨する。既存ドキュメントに明示的なデプロイ構成の記載がないため、`ops.md` のセットアップセクションから要約を引用する程度でよい。

**[P1-3] architecture.md の「データの入出力形式」の配置先の妥当性**
- **対象セクション**: 5.2 新ドキュメントの構成（docs/architecture.md「データフロー」セクション）
- **内容**: `arch.md` の「データの入出力形式」（テーブル: レース情報→SQLite、学習入力→CSV 等）を `architecture.md` の「データフロー」セクションに配置する計画となっている。この情報はデータの物理的な格納形式であり、Data ビューポイント（`domain-model.md`）の「データ永続化」セクションに配置する方がIEEE 1016 の分類として自然である可能性がある。ただし、データフローの文脈で入出力形式を併記することにも合理性があるため、最終的には実装者の判断に委ねてよい。
- **修正の期待値**: 実装時に `architecture.md`（Structure 視点: 技術的なデータフローの中間形式）と `domain-model.md`（Data 視点: 永続化方式）のどちらが適切か検討することを推奨する。現状の方針のままでも大きな問題はない。
